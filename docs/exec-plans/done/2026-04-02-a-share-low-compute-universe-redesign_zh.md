# 执行计划：A股低算力股票池重构

- 状态：已于 2026-04-03 完成

## 目标

定义并在后续实现一套更稳定、更确定、且受算力约束的 A 股 OHLCVA 选股股票池规则，使得：

- 训练仍然能在有限硬件上完成
- 预测结果可复现，不依赖随机抽样
- 换手控制通过显式持仓规则完成，而不是依赖临时性的股票池合并
- 后续回测可以在同一套验证口径下比较现有方案与重构方案

## 范围

- 为以下内容制定研究与实施计划：
  - 在 `model_function/` 下沉模型域股票池共享契约
  - 月度滞后训练股票池构建
  - 月度滞后预测股票池构建
  - 显式持仓缓冲区 / 容量控制规则
  - 训练与预测路径共用的股票池工具逻辑
  - 定向测试与文档更新
- 已结合现行代码与测试确认的实际实现位置：
  - `model_function/universe.py`
  - `runtime/adapters/modeling.py`
  - `scripts/filter.py`
  - `scripts/build_portfolio.py`
  - `test/` 下相关测试
  - `README.md`、`docs/README_zh.md`、`ARCHITECTURE.md`、`docs/ARCHITECTURE_zh.md` 以及 `docs/navigation-docs/*`

不在范围内：

- 引入 OHLCVA 之外的新特征
- 修改模型结构、标签定义或训练框架
- 日内执行仿真或券商接入
- 重写更大范围的 runtime 流程

## 假设

- 项目仍然聚焦于 A 股日频选股。
- 防未来函数必须严格保留：任意 `t` 月的股票池成员资格，只能由 `t-1` 月末之前可获得的数据决定。
- 除非后续任务明确修改，否则现有的 ST 剔除与指数剔除规则继续保留。
- 有限硬件仍然是硬约束，因此训练与预测工作量都必须保留明确上限。
- 在重构方案实现并验证前，既有月度股票池逻辑仍然作为 incumbent baseline 保留。
- 由于本计划是在“尽量少看代码”的约束下形成的，文件归属、指标可用性、验证能力等假设在实施前都必须通过代码与测试再次确认。

## 目标规则契约

下面的数值带宽与指标选择，只是后续评估用的起始假设，不应视为已经定稿的实施标准。它们的作用是让后续工作更具体、可测试，但在真正落地前必须结合现行代码、可用数据字段和 runtime 归属再次确认。

1. 拆分三层决策。
   - 训练股票池回答：哪些股票允许参与模型拟合。
   - 预测股票池回答：今天哪些股票需要打分。
   - 持仓缓冲区回答：已有持仓在排名下滑后哪些仍可继续持有。

2. 仅在训练降采样中保留随机性。
   - 训练阶段可使用分层降采样以适配算力限制。
   - 预测阶段在相同日期快照和配置下必须完全确定。

3. 使用对异常放量更稳健的滞后流动性指标。
   - 优先使用 20 到 60 个交易日窗口上的滚动成交额中位数。
   - 如果项目里没有直接成交额字段，则使用已有收盘价与成交量推导。

4. 用显式缓冲区替代临时性的前一交易日预测结果合并逻辑。
   - 可先按以下目标口径评估：
     - 预测入池带：按滞后流动性取前 1000
     - 预测留任带：按滞后流动性取前 1200
     - 新买入带：按模型排名取前 300
     - 持有带：按模型排名取前 500
   - 最终阈值先视为暂定，待验证后再固化。

5. 训练股票池应宽于预测股票池。
   - 可先按以下目标口径评估：
     - 训练入池带：按滞后流动性取前 1800
     - 训练留任带：按滞后流动性取前 2200
     - 如果整池训练仍过重，则再以确定性的分层采样降到 800 只

## 步骤

1. 在真正改行为前，先确认实际实现归属。
   - 当实施开始时，按模块变更任务重新走一次 navigation path。
   - 确认现行归属到底在 `scripts/*`、runtime 适配层、共用工具模块，还是其他位置。
   - 确认当前用于定义股票池行为的测试与产物。
   - 在这一步完成前，不应把本计划里暂定的文件/模块列表视为权威事实。

2. 冻结现有行为与对比口径。
   - 把当前训练池、预测池、前一交易日合并规则整理成 baseline contract。
   - 记录当前月度股票池规模、预测日候选规模、最终持仓容量假设。
   - 把冻结后的 baseline 放到一份专门的中英双语契约文档中：
     - `docs/product-specs/a-share-universe-contract.md`
     - `docs/product-specs/a-share-universe-contract_zh.md`
   - 明确当前痛点：
     - 预测资格中存在随机性
     - 股票池筛选与换手控制职责混杂
     - 训练分布与预测分布不一致

3. 在同一对双语文档里定义重构后的股票池契约。
   - 明确滞后口径、流动性指标、缓冲区语义、确定性采样规则。
   - 明确哪些步骤允许随机，如何保证可复现。
   - 在正式切换完成前，把 incumbent 与重构方案并列记录在同一对文档中。
   - 保持契约足够小，能够仅通过输出文件进行测试。

4. 在改行为前，先抽取共用股票池逻辑。
   - 统一训练与预测路径的共用过滤和排序逻辑。
   - 共用路径应支持：
     - 滞后流动性排序
     - 训练阶段确定性分层采样
     - 预测阶段确定性的入池/留任缓冲区
     - 指数与 ST 剔除钩子

5. 实现训练股票池重构。
   - 保留月度刷新与防未来函数语义。
   - 只有在新的确定性契约已经落地后，才替换既有分组逻辑。
   - 如果仍需降采样，必须通过固定 seed 或日期/股票哈希规则保证可复现。

6. 实现预测股票池与持仓缓冲区重构。
   - 去掉预测阶段的随机压缩逻辑。
   - 用显式持有带规则替代前一交易日预测结果合并。
   - 最终容量上限必须明确且可测试。

7. 补充定向验证。
   - 增加以下单元测试：
     - 滞后是否正确
     - 重复运行输出是否一致
     - 入池/留任缓冲区行为
     - 容量上限行为
     - 指数与 ST 是否泄漏
   - 至少增加一个固定快照上的回归样例，对比 incumbent 与重构方案的输出面。

8. 在切换前做最小必要验证。
   - 对比 incumbent 与重构方案在以下维度的表现：
     - 股票池稳定性
     - 预测日候选切换率
     - 持仓换手代理指标
     - 最终股票数量
     - 仅在确认仓库确实能稳定产出此类证据后，才纳入排名/回测摘要比较
   - 如果新方案虽然提升了确定性，但明显伤害可交易性或下游信号质量，则不要切换。

9. 更新文档与追踪产物，完成后再迁移计划。
   - 如果操作层可见的股票池规则发生变化，更新 `README.md` 与 `docs/README_zh.md`。
   - 在重构进行中，本计划持续放在 `docs/exec-plans/active/`。
   - 只有在实现和验证都完成后，才移动到 `docs/exec-plans/done/`。

## 验收标准

- 训练、预测、持仓缓冲区三层逻辑都有成文且中英双语一致的规则契约。
- 相同日期快照与配置下，预测股票池成员资格完全确定。
- 如果训练仍需降采样，其规则必须显式可复现，并与预测逻辑分离。
- 换手控制通过显式持有带 / 容量规则表达，而不是依赖临时性的前一交易日预测结果合并。
- 滞后流动性排序仍然满足防未来函数，并兼容月度刷新。
- 在行为变更合入前，必须先通过现行代码确认实现归属与测试覆盖面。
- 定向测试覆盖滞后行为、确定性、剔除规则、缓冲区逻辑、容量上限。
- 切换前必须保留 incumbent 与重构方案的对比记录。

## 回滚说明

- 在新的股票池契约通过定向验证前，保留既有月度股票池逻辑。
- 如果重构带来更高不稳定性、隐藏的未来函数风险或下游回归问题，则回滚行为修改，但保留仍可复用的工具抽取和与 incumbent 一致的文档更新。
- 如果验证显示阈值对当前硬件或市场覆盖过于激进，则保留三层契约结构，只调整具体数值带宽。
