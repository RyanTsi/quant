# 执行计划：Phase 3 交易日与执行上下文规范化

## 目标

统一模型流水线中的日期处理，让预测与组合构建阶段使用一致的交易日历语义，并把剩余的日期默认行为收口到更清晰的执行上下文边界。

## 范围

- `runtime/adapters/modeling.py`
- `runtime/services.py`
- 如有需要，修改 `model_function/qlib.py` 或新增一个小型共享 helper 模块
- 聚焦的模型/运行时测试
- 直接受影响的运行时与架构文档

## 假设

- Phase 1 与 Phase 2 已建立的 helper 边界和运行状态边界继续作为事实来源。
- 除非确定性重跑契约必须要求显式日期，否则对外 CLI 契约保持稳定。
- 当前最小且有效的修改是集中处理交易日解析，而不是重写整个模型流水线。
- 预测池连续性和组合连续性都应从同一套交易日历中解析“前一状态”产物。

## 步骤

1. 引入共享的交易日 / 执行上下文 helper 边界。
   - 统一处理：
     - 交易日选择
     - 前一交易日查找
     - 默认执行日期的回退行为

2. 让预测与组合路径都使用该共享边界。
   - 保持预测池上一期持仓查找继续采用交易日历语义。
   - 将组合构建中按自然日查找上一期目标文件的逻辑替换为同样的交易日历语义。

3. 在需要的地方收紧确定性日期默认行为。
   - 移除或缩小那些本应优先使用显式日期或共享运行时日期解析器的 wall-clock 回退。
   - 若某些路径本就有意使用“本地最新交易日”作为默认值，则保留当前操作体验。

4. 增补聚焦测试。
   - 覆盖周末/节假日连续性行为。
   - 覆盖当请求日期不在本地交易日历中时，共享 helper 的失败语义。
   - 如果引入新的执行上下文参数，覆盖 service/adapter 契约。

5. 更新文档与追踪产物。
   - 保持中英文文档对交易日语义与重跑行为的描述一致。

## 验收标准

- 预测与组合两个阶段的上一状态查找都使用一致的“前一交易日”语义。
- 模型流水线不再在相邻的预测/组合阶段混用交易日和自然日的上一文件解析方式。
- 共享 helper 的测试能让日期契约可审查，而无需重新打开整个 adapter 热点文件才能理解行为。
- 运行时文档准确描述修改后的默认日期行为。

## 验证

- 聚焦模型/运行时测试：
  - `test/test_predict_pool.py`
  - `test/test_modeling_adapter_contract.py`
  - `test/test_model_pipeline_service.py`
- 任何新增的交易日 helper 或执行上下文行为测试

## 回滚说明

- 在可能的情况下保持对外入口签名稳定。
- 如果共享 helper 带来的改动面过大，则允许 helper 先留在 adapter 内部，但必须保留统一后的交易日语义和直接测试。
